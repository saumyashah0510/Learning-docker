Docker is a platform that lets you package an application along with everything it needs to run — code, runtime, system libraries, settings — into a single unit that runs the same way on any machine. It solves the classic "it works on my machine" problem.Before Docker, if you wrote an app on your laptop, it might behave differently on a coworker's laptop or on a production server because of differences in OS versions, installed libraries, or configurations. Docker fixes this by bundling everything the app needs into an isolated package.It does this using containers — a lighter-weight alternative to running full virtual machines.

__________________________________________________________________________________________________________________________________________________________

Image :
A Docker image is a read-only blueprint or template for your application. It contains the application code, a stripped-down OS layer, runtime (like Node.js or Python), libraries, and configuration files. Think of it like a recipe or a snapshot — it doesn't run by itself, it just describes what should exist.

Container
A container is a running instance of an image. If the image is the recipe, the container is the actual dish you cooked from it. You can start, stop, move, or delete a container, and you can spin up multiple containers from the same image (e.g., 5 containers all running the same web app image, perhaps to handle more traffic).

    # Add Docker's official GPG key:
    sudo apt update
    sudo apt install ca-certificates curl
    sudo install -m 0755 -d /etc/apt/keyrings
    sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
    sudo chmod a+r /etc/apt/keyrings/docker.asc

    # Add the repository to Apt sources:
    sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
    Types: deb
    URIs: https://download.docker.com/linux/ubuntu
    Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
    Components: stable
    Architectures: $(dpkg --print-architecture)
    Signed-By: /etc/apt/keyrings/docker.asc
    EOF

    sudo apt update

Inside container : 
    apt update
    Refreshes the local package index — it contacts the package repositories listed in the container's sources and downloads the latest list of available packages and their versions. It doesn't install or upgrade anything itself; it just updates the "catalog."

    apt upgrade
    Looks at that refreshed catalog and actually installs newer versions of any packages currently installed in the container, if updates are available.

Any changes you make inside a running container — installing packages, upgrading software, creating files — only exist in that container's writable layer. They are not saved into the image, and they will vanish:
If the container is removed (docker rm)
If you spin up a new container from the same image (the new one starts fresh, ignoring whatever you changed in the old one)

Docker can build images automatically by reading instructions from a dockerfile.
The command "docker build" reads the instructions in a docker build and creates a docker image. 

DOCKER FILE STRUCTURE

Docker image consists of layers
Each layer maps to a docker instruction in a docker file.

__________________________________________________________________________________________________________________________________________________________

From : Set the base image for the docker file.
    FROM <image>[:tag]
    Example : FROM ubuntu

__________________________________________________________________________________________________________________________________________________________

Run : runs one or more commands    
It is possible to perform multiple commands with a single RUN statement
    Example : RUN apt-get update && apt-get install -y \vim \git

__________________________________________________________________________________________________________________________________________________________

docker history <image_name> : To display image layers and their details

__________________________________________________________________________________________________________________________________________________________

CMD : Used to specify the command to be executed when the container is started
Only one CMD command should exist in a dockerfile
If more than one only the latest command will take effect.

    CMD["executable" , "param1", "param2"..]
    Example : CMD["echo","Hello World"], CMD ["Python" , "app.py"] (Executes "python app.py" at container start)

__________________________________________________________________________________________________________________________________________________________

docker build [OPTIONS] PATH | URL | -
    PATH : The build context - usually the current directory
    Example : docker build -t myapp
By default docker looks for a file names Dockerfile but you can specify a new one.

-- Relative paths : docker build -f files/test.txt image3 /imagefiles :

    docker build
    The base command that tells Docker to build an image from a Dockerfile.

    -f files/test.txt
    The -f flag specifies which file to use as the Dockerfile, instead of the default behavior.

    Normally, docker build assumes your Dockerfile is named exactly Dockerfile and sits in the root of your build context. Here, that default is overridden:

    The file is named test.txt (not Dockerfile)
    It lives inside a subfolder called files/
    So Docker will read its build instructions from files/test.txt, treating it as if it were a Dockerfile — the filename itself doesn't matter to Docker; what matters is that the content follows Dockerfile syntax (FROM, RUN, COPY, etc.).

    -t image3
    The -t (tag) flag names the resulting image. Once built, this image will be tagged as image3 (equivalent to image3:latest since no version tag was given).

    /imagefiles
    This is the build context — the folder Docker will use as the root directory for the build.

    This is an important and often-misunderstood part: the build context is the set of files Docker has access to during the build (for things like COPY or ADD instructions). Docker will only be able to copy in files that live inside /imagefiles, even though the Dockerfile itself is being read from files/test.txt.

    Putting it together
    This command means:

    "Build an image named image3, using the build instructions found in files/test.txt, and use /imagefiles as the folder of files available to copy into the image."

    Important subtlety: where is files/test.txt relative to?
    This trips people up. The path to -f is normally resolved relative to the build context, not relative to your current working directory.

    So in this case, Docker would look for the Dockerfile at:

    /imagefiles/files/test.txt
    ...not at a files/test.txt folder relative to wherever you're running the command from. If files/test.txt doesn't actually exist inside /imagefiles, this build will fail with a "no such file or directory" error.


-- Absolute paths : docker build -f /home/ubuntu/apps/imagefiles/test.txt -t image3/ home/ubuntu/build/contextfiles/app1    

    This is showing you that the Dockerfile and the build context don't have to be in the same place at all.

    The Dockerfile (test.txt) lives in: /home/ubuntu/apps/imagefiles/
    The build context lives in: /home/ubuntu/build/contextfiles/app1

    So if test.txt contains something like:
    FROM ubuntu:22.04
    COPY app.py /app/app.py
    Docker will look for app.py inside /home/ubuntu/build/contextfiles/app1/, not anywhere near where test.txt itself is sitting.

EXPOSE : informs docker that the container listens on the specified networks ports at runtime
    EXPOSE <port> 
    Example : EXPOSE 80 443 (Exposes port 80 and 443 (HTTP and HTTPS))
It is informational only and does not actually publish or expose the port 

__________________________________________________________________________________________________________________________________________________________

COPY : copies new files or directories from <src> and adds them to the filesystem of the container at the path <dest>
    COPY <src> <dest>
It can copy files only from the build context.    

__________________________________________________________________________________________________________________________________________________________

docker run -p 80:80 image_name : It maps port 80 of our device withport 80 of the container(if exposed).

__________________________________________________________________________________________________________________________________________________________

WORKDIR : Used to define the working directory of the docker container at any given time
    WORKDIR </path>

    BEFORE :-
        FROM node:18
        COPY app.js /usr/src/app/app.js
        COPY package.json /usr/src/app/package.json
        RUN cd /usr/src/app && npm install
        CMD cd /usr/src/app && node app.js

    AFTER :-
        FROM node:18
        WORKDIR /usr/src/app
        COPY app.js .
        COPY package.json .
        RUN npm install
        CMD ["node", "app.js"]    

You can use WORKDIR multiple times
Each call changes the "current folder" for everything after it:
